# S12 — Editor Integrations

| Field | Value |
| --- | --- |
| Status | Not started |
| Depends on | [S9](S9-command-expansions.md) |
| ADRs | [ADR-0008](../adr/0008-editor-extension-packaging.md), [ADR-0009](../adr/0009-xtask-manifest-generator.md) |

## Goal

Package `snipper-lsp` as a first-class editor extension for VS Code and Visual Studio so that
all five expansion types are available in real editors without any manual server configuration.

## Strategy

The Rust backend (`snipper-core`, `snipper-context`, `snipper-lsp`, `snipper-cli`) remains the
single source of truth for all expansion logic. IDE extensions are thin wrappers with exactly
four responsibilities: locate the binary, start the process, pass settings, and handle
IDE-specific insertion (snippet tabstop activation). No matching, rendering, ranking, or
TOML rule logic must appear in any extension.

## Required server changes before S12.1

### Fix execute_command

The S9 handler currently applies the edit via `workspace/applyEdit`, which carries no
`InsertTextFormat::SNIPPET` field. Tabstop markers (`${1:ClassName}`, `$0`) are inserted as
literal text (see [S9 known limitations](S9-command-expansions.md)).

The handler must be changed to return the body as a result value:

```rust
// Before (S9):
self.client.apply_edit(edit).await.ok();
Ok(None)

// After (S12 prerequisite):
Ok(Some(serde_json::Value::String(rule.body.clone())))
```

The extension then calls `workspace/executeCommand`, receives the body string, and inserts it
via the IDE snippet API (see subtask details below).

### Read roslynPath from initializationOptions

The Roslyn sidecar path must be configurable from the IDE without requiring an env var.
`snipper-lsp` must read `roslynPath` (and optionally `trace`) from the LSP
`initializationOptions` sent by the client on `initialize`. Env var `SNIPPER_ROSLYN` remains
as the lowest-priority fallback (see [Settings hierarchy](#settings-hierarchy)).

## Subtasks

### S12.1 — VS Code extension

Build a TypeScript extension using `vscode-languageclient`.

#### Layout

```text
extensions/snipper-vscode/
  package.json                  ← human-authored; contributes, activationEvents, config schema
  package.generated.json        ← machine-generated by xtask; command list only (see ADR-0009)
  src/
    extension.ts                ← activate / deactivate; wires up client
    serverPath.ts               ← binary locator: setting → bundled → PATH
    commands.ts                 ← registers snipper.* commands; calls executeCommand; insertSnippet
  test/
  bin/
    win32-x64/snipper-lsp.exe
    linux-x64/snipper-lsp
    darwin-arm64/snipper-lsp
  sidecar/
    win32-x64/Snipper.Roslyn.exe
```

#### Activation

Activate only on C# documents and on each `snipper.*` command to avoid unnecessary startup:

```json
"activationEvents": [
  "onLanguage:csharp",
  "onCommand:snipper.scaffoldConstructor",
  "onCommand:snipper.scaffoldProperty",
  "onCommand:snipper.implementInterface"
]
```

#### Settings contribution

```json
"configuration": {
  "properties": {
    "snipper.serverPath": {
      "type": "string",
      "default": "",
      "description": "Path to snipper-lsp binary. Empty = use bundled binary."
    },
    "snipper.roslynPath": {
      "type": "string",
      "default": "",
      "description": "Path to Snipper.Roslyn sidecar. Empty = disable Roslyn enrichment."
    }
  }
}
```

#### Binary locator (`serverPath.ts`)

Resolution order: `snipper.serverPath` setting → bundled `bin/{platform}/snipper-lsp[.exe]` →
first `snipper-lsp` on `PATH`. Failure at each level falls through; final failure throws a
user-visible error.

#### Command insertion (`commands.ts`)

Call `workspace/executeCommand`, receive the body string, apply via `editor.action.insertSnippet`
to activate tabstops:

```typescript
const body = await client.sendRequest('workspace/executeCommand', {
  command: 'snipper.scaffoldConstructor',
  arguments: [{
    textDocument: { uri: editor.document.uri.toString() },
    position: editor.selection.active
  }]
}) as string;

await vscode.commands.executeCommand('editor.action.insertSnippet', {
  snippet: new vscode.SnippetString(body)
});
```

**Inputs:** `snipper-lsp` binary (S4 + S12 server fix); command registry (S9).

**Outputs:** `extensions/snipper-vscode/` TypeScript package; `snipper-*.vsix` artefact.

### S12.2 — Visual Studio extension

Build a C# VSIX extension using `ILanguageClient`.

#### Layout

```text
extensions/snipper-vs/
  Snipper.VisualStudio.sln
  Snipper.VisualStudio/
    SnipperLanguageClient.cs    ← ILanguageClient; starts snipper-lsp; handles restarts
    SnipperOptionsPage.cs       ← Tools → Options page; serverPath + roslynPath settings
    SnipperCommandPackage.cs    ← AsyncPackage; registers VS commands on load
    SnipperCommandRouter.cs     ← handles snipper.* commands; calls executeCommand; inserts
    SnipperBinaryLocator.cs     ← options setting → bundled → PATH
    Generated/
      SnipperCommands.cs        ← machine-generated by xtask (see ADR-0009)
    bin/
      snipper-lsp.exe
    sidecar/
      Snipper.Roslyn.exe
```

Visual Studio is Windows-only; the initial release targets Windows x64 only.

#### Activation

Activate on `ContentType("CSharp")`; also handle `.csproj` for project-level Roslyn workspace
loading.

#### Settings

`SnipperOptionsPage` exposes `ServerPath` and `RoslynPath` string properties stored in the VS
settings store. These are passed to `snipper-lsp` via `initializationOptions` on
`ILanguageClient.LoadAsync`.

#### Command insertion (`SnipperCommandRouter`)

Call `workspace/executeCommand` via `ILanguageClient.RequestAsync`, receive the body string,
insert via `IVsExpansionManager` or `IOleCommandTarget` snippet insertion to activate tabstops.

**Inputs:** `snipper-lsp` binary (S4 + S12 server fix); command registry (S9).

**Outputs:** `extensions/snipper-vs/` C# project; `Snipper.VS.vsix` artefact.

### Deferred: other editors

Neovim, Emacs, JetBrains Rider, and other LSP-capable editors can connect to `snipper-lsp`
with user-side configuration (`:LanguageClient-neovim`, `lsp-mode`, etc.). Dedicated wrappers
are deferred until community demand is established.

## xtask manifest generator

Command metadata must not be maintained in two places. A Rust `xtask` crate reads
`snippets/csharp/commands.toml` and generates the command contributions for both extensions
(see [ADR-0009](../adr/0009-xtask-manifest-generator.md)):

```text
cargo run -p xtask -- generate-extension-manifests
```

Outputs:

- `extensions/snipper-vscode/package.generated.json` — VS Code command contributions fragment,
  merged into `package.json` at build time.
- `extensions/snipper-vs/Generated/SnipperCommands.cs` — C# constants for command IDs and
  titles, consumed by `SnipperCommandPackage` and `SnipperCommandRouter`.

CI must verify that generated files are up to date (run `xtask generate-extension-manifests`
and fail if `git diff --exit-code` is non-empty).

## Settings hierarchy

Three levels, resolved in priority order:

| Priority | Source | Example |
| --- | --- | --- |
| 1 (highest) | IDE setting | `snipper.roslynPath` in VS Code settings.json |
| 2 | LSP `initializationOptions` | `{"roslynPath": "/usr/local/bin/Snipper.Roslyn"}` |
| 3 (fallback) | Environment variable | `SNIPPER_ROSLYN=/usr/local/bin/Snipper.Roslyn` |

`initializationOptions` also accepts `"trace": "off" | "messages" | "verbose"` to enable
LSP message tracing without modifying the server binary.

## Binary packaging

Release builds bundle platform-specific binaries alongside the extension. Users must not
need a Rust toolchain or .NET SDK installed.

```text
extensions/snipper-vscode/bin/
  win32-x64/snipper-lsp.exe
  linux-x64/snipper-lsp
  darwin-arm64/snipper-lsp
extensions/snipper-vscode/sidecar/
  win32-x64/Snipper.Roslyn.exe    ← Roslyn sidecar (Windows only for V1)

extensions/snipper-vs/
  bin/snipper-lsp.exe             ← Windows x64 only for V1
  sidecar/Snipper.Roslyn.exe
```

Platform detection (`win32`, `linux`, `darwin` + arch) lives in the extension binary locator,
not in `snipper-lsp`. Dev builds may omit bundled binaries and rely on PATH.

## Test layers

Four layers, each testing a different scope:

### Layer 1 — Rust unit and integration tests

Existing `cargo test --workspace`. Must add:

- `execute_command` handler returns `Value::String(body)` — not `None`.
- No call to `apply_edit` for command rules.

### Layer 2 — LSP black-box tests

Spawn `snipper-lsp` as a child process; drive it over stdio JSON-RPC:

```text
→ initialize
← InitializeResult (with supportedCommands list)
→ textDocument/didOpen  (.cs fixture)
→ textDocument/completion  (position after "users.fo")
← CompletionList (contains "fod" candidate)
→ workspace/executeCommand  { command: "snipper.scaffoldConstructor", arguments: [...] }
← "public ${1:ClassName}()\n{\n    $0\n}"   ← must be a string result, not null
```

These live in `tests/lsp_blackbox/` and run in CI via `cargo test`.

### Layer 3 — VS Code extension integration tests

Using `@vscode/test-cli` and `@vscode/test-electron` with an Extension Development Host:

- Opening a `.cs` fixture activates the extension.
- Opening a `.rs` or `.py` file does not activate the extension.
- Invoking "Snipper: Scaffold constructor" inserts the stub with tabstop navigation active
  (first tabstop selected, not literal `${1:ClassName}` text).

### Layer 4 — Visual Studio integration tests

Using `microsoft/vs-extension-testing` (experimental Visual Studio instance):

- Opening a `.cs` file activates `SnipperLanguageClient`.
- Completion request returns Snipper candidates.
- Invoking a command from the VS command palette inserts the stub with tabstop navigation.

## Order of work

1. Fix `execute_command` to return snippet body string; add Layer 1 tests.
2. Write Layer 2 LSP black-box test suite; verify the fix end-to-end.
3. Create `extensions/snipper-vscode/` scaffold (`package.json`, `tsconfig.json`).
4. Wire up `vscode-languageclient`; verify `initialize` handshake.
5. Implement `serverPath.ts` binary locator (setting → bundled → PATH).
6. Run `xtask generate-extension-manifests`; merge generated commands into `package.json`.
7. Implement `commands.ts`: call `workspace/executeCommand`, insert via
   `editor.action.insertSnippet`.
8. Write Layer 3 VS Code integration tests; verify tabstop activation.
9. Create `extensions/snipper-vs/` scaffold.
10. Implement `SnipperLanguageClient`, `SnipperBinaryLocator`, `SnipperOptionsPage`,
    `SnipperCommandRouter`; write Layer 4 tests.

## What not to do

- **No engine logic in extensions.** Matching, ranking, TOML parsing, and template rendering
  must never appear in `extension.ts` or any C# VSIX class. The server is authoritative.
- **No transpiler.** Do not generate the Visual Studio extension from the VS Code extension.
  They share protocol and generated manifests; UI wiring is IDE-specific and must stay so.
- **No Roslyn bindings in Rust.** The sidecar model (ADR-0007) is the correct boundary.
  Tree-sitter provides CST; Roslyn provides C# semantics. Do not collapse the boundary.
- **No rich UI on first iteration.** Completion, surround, command palette, settings, restart.
  Tool windows and status bar items come after the core loop is proven.

## Inputs → Outputs

**In:** `snipper-lsp` binary (S4); `workspace/executeCommand` handler (S9).

**Out:** updated `execute_command` returning body string; `extensions/snipper-vscode/`;
`extensions/snipper-vs/`; `xtask generate-extension-manifests`; marketplace `.vsix` artefacts.

## Acceptance criteria

### S12.1 VS Code

- Installing the VSIX and opening a `.cs` file triggers Snipper completions
  (postfix, prefix, surround) without any manual configuration.
- "Snipper: Scaffold constructor" inserts the constructor stub at the cursor and activates
  tabstop navigation (`${1:ClassName}` is selected; `$0` is the final cursor position).
- Tabstop markers are never inserted as literal text.
- Extension activates only on C# documents; no activation on `.rs`, `.py`, etc.
- `snipper.serverPath` overrides the bundled binary path.
- `snipper.roslynPath` enables Roslyn enrichment when set.
- Extension handles `snipper-lsp` crash by restarting the server automatically.
- Layer 3 integration tests pass in CI.

### S12.2 Visual Studio

- Installing the VSIX and opening a `.cs` file triggers Snipper completions.
- "Snipper: Scaffold constructor" inserts the stub and activates tabstop navigation.
- Tabstop markers are never inserted as literal text.
- Extension activates only on C# documents.
- Options page exposes `ServerPath` and `RoslynPath` overrides.
- Layer 4 integration tests pass in CI.

### Shared

- `xtask generate-extension-manifests` is idempotent; CI diff check passes.
- LSP Layer 2 black-box test: `workspace/executeCommand` returns a non-null string result.
